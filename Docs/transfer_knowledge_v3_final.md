# 📦 Transfer Knowledge File (v3 — FINAL)

## 0. Назначение
Единый документ для переноса проекта балансирующего робота (STM32 + ODrive + IMU + Python GUI) без потери контекста: архитектура, протокол, регулятор, коэффициенты (с диапазонами), ошибки, наблюдения, логи, AI-интеграция, и пошаговая методика настройки.

---

## 1. Архитектура системы

### 1.1 Узлы
- **STM32 (F401/F405)** — контур управления (PID-подобный/LQR-лайт), USB CDC лог/CLI
- **ODrive v3.6** — torque control, энкодеры (pos/vel)
- **IMU ICM20948** — pitch, pitch_rate (SPI + DMA + EXTI)
- **Python GUI (v9.0.4+)** — управление параметрами, лог, AI-анализ

### 1.2 Потоки данных
- IMU → EKF/оценка → pitch, rate
- ODrive → CAN → pos/vel → wheel_vel
- STM32 → управление → torque (ODrive)
- STM32 → USB CDC → DBG строки
- GUI ↔ STM32 → команды get/set/en

---

## 2. Протокол

### 2.1 Команды
```
get all
set <name> <value>
en 0
en 1
```

### 2.2 Формат лога
```
DBG t=... en=... valid=... mode=IDLE|CATCH|BALANCE pitch=... rate=... P=... D=... V=... raw=... clamp=... sat=... wv=... wd=... wp=... wpr=... ... u=... L=... R=...
```

Ключевые поля:
- `en` — включен ли контроль
- `mode` — режим
- `pitch` (мрад) — угол
- `rate` (мрад/с) — угловая скорость
- `wv` — средняя скорость колес
- `raw/clamp/sat` — до/после ограничений

---

## 3. Регулятор (фактический)

Базовая форма:
```
u = Kp * pitch + Kd * rate + Kv * wheel_vel + Usync
u = clamp(u, ±CONTROL_U_LIMIT)
```

Где:
- `Kp = control_k_pitch`
- `Kd = control_k_pitch_rate`
- `Kv = control_k_wheel_vel`
- `Usync` — синхронизация колес (малый вклад)

Режимы:
- **IDLE**: u = 0
- **CATCH_FROM_TILT**: усиленный Kp/Kd, ограничение `catch_u_limit`
- **BALANCE**: основной контур

## 5. Критические проблемы и причины

### 5.1 Смещение нуля pitch (главная)
Симптом:
```
IDLE: pitch ≈ 0.8–1.0 rad
```
Вывод: неверная «вертикаль» → BALANCE почти недостижим.

### 5.2 Нет устойчивого BALANCE
- кратковременная стабилизация
- затем раскачка → рост скорости базы → падение

Причина: нет достаточного демпфирования по скорости/накоплению.

### 5.3 Saturation
```
sat=1 при clamp≈limit
```
Причина: либо не хватает момента, либо неправильные коэффициенты.

### 5.4 Шум/спайки rate
Скачки до 400–600+ мрад/с → ложные реакции.

## 6. Наблюдения

- Робот **иногда стоит вертикально** → базовая логика верна
- Основная проблема — **устойчивость во времени**
- CATCH снижает угол, но **переход в BALANCE нестабилен**
- Сильная чувствительность к `control_pitch_trim_rad`

---

## 7. Антипаттерны (НЕ ДЕЛАТЬ)

- Увеличивать `Kp` > ~1.0 (ухудшает)
- Большой `trim` (ломает рабочую точку)
- Слишком агрессивный CATCH (выбрасывает в IDLE)
- Игнорировать фильтрацию `rate`

---

## 8. Что работает

- Умеренный `Kp ~0.8`
- `Kd ~0.45–0.5`
- Небольшой `Kv`
- Ограничение момента (U_LIMIT)
- Мягкий CATCH + гистерезис

---

## 9. Типовые логи (эталоны)

### 9.1 Успешный CATCH (частично)
```
en=1 mode=CATCH pitch: 0.7 → 0.1 rad
```

### 9.2 Проблемный случай
```
pitch падает, затем растет до >0.8 rad → IDLE
```

### 9.3 Неверный ноль
```
IDLE pitch ≈ 0.85 rad
```

## 14. Пошаговая методика настройки

1. **Ноль pitch**
   - добиться IDLE: pitch ≈ 0
   - через калибровку/trim

2. **CATCH**
   - обеспечить уверенное снижение угла
   - без saturation

3. **BALANCE**
   - включается при малом pitch/rate
   - не вываливается сразу

4. **Демпфирование**
   - увеличить Kd/Kv при раскачке

5. **Гистерезис режимов**
   - настроить catch↔balance пороги
