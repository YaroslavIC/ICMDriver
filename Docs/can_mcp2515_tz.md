# ТЗ и накопленные знания для нового драйвера `can_mcp2515`

Дата подготовки: 08.03.26 14:20:00

## Назначение

Подготовить **новый** драйвер `can_mcp2515` для проекта `ICMDriver` на STM32F401CCU6.

Драйвер нужен для работы с модулем:
- CAN controller: **MCP2515**
- CAN transceiver: **TJA1050**

На текущем этапе драйвер должен быть простым, устойчивым и удобным для отладки.

Цель новой версии:
1. Сначала получить **надёжный приём heartbeat** от двух ODrive.
2. После подтверждения устойчивой работы добавить приём `Encoder Estimates`.
3. Только после этого обсуждать активные запросы, scheduler и DMA.

Новый базовый файл драйвера:
- `can_mcp2515.h`
- `can_mcp2515.c`

## Критически важные выводы из предыдущей отладки

### 1. Причина главной проблемы уже найдена
Модуль `MCP2515 + TJA1050` **не работал без 5 В питания**.

Фактически подтверждено:
- без 5 В на модуле MCP2515/TJA1050 не было приёма CAN-кадров;
- после подачи 5 В появились:
  - прерывания `INT`,
  - принятые heartbeat,
  - валидные ID и DLC.

### 2. Минимальный тестовый драйвер доказал работоспособность железа
После подачи 5 В были подтверждены:
- прерывания по `INT` работают;
- heartbeat от ODrive приходят;
- последний принятый ID был `0x41`;
- `DLC = 8`;
- счётчики прерываний и heartbeat росли.

### 3. Старый сложный драйвер не брать за основу как есть
Причины:
- слишком много логики сразу;
- было трудно отделять ошибки кода от ошибок железа;
- были scheduler, DMA, loopback, timestamp, FIFO sample, heartbeat и encoder в одной структуре.

Новый драйвер нужно строить **поэтапно** от минимально рабочего варианта.

## Актуальные требования пользователя

### Общие правила кода
Основаны на `code_rules_f401ccu.md`, но с учётом уточнений пользователя.

Обязательно:
- использовать **HAL**;
- все функции драйвера разместить в **одном c/h модуле**;
- все данные, включая параметры инициализации и возвращаемые данные, хранить **в одной структуре** и передавать только через неё;
- использовать стиль C99;
- не использовать динамическую память;
- все публичные функции должны возвращать **только статус выполнения**;
- проверять входные параметры;
- проверять результаты HAL-вызовов;
- `#include "main.h"` в заголовочном файле;
- использовать include guard;
- константы из документации выносить в `#define`;
- инициализацию выполнять через константы из `.h`;
- код должен быть коротким и лаконичным;
- не плодить мелкие редкие функции без необходимости;
- в `.h` сразу после заголовка должен быть **пример использования**.


**Новое актуальное требование пользователя:**
- комментарии в коде делать в **UTF-8**.

### Ограничение на комментарии
- не использовать последовательности `/*` и `*/` в коде и примерах;
- все комментарии оформлять только через `//` в начале строки.

### Версионирование
В заголовке `.h` файла обязательно указывать:
- версию драйвера в формате **`x.y`**;
- дату и время сборки в формате **`dd.mm.yy hh:mm:ss`**.

Пример:
- Версия драйвера: `0.21`
- Дата сборки: `08.03.26 14:20:00`

Также в `.h` нужны defines вида:
- `CAN_MCP2515_DRIVER_VERSION_STR`
- `CAN_MCP2515_DRIVER_BUILD_STR`

## Аппаратная платформа

### Микроконтроллер
- STM32F401CCU6
- Проект: `ICMDriver`
- Частота CPU: **84 МГц**

### Пины подключения MCP2515
Подтверждённая разводка:
- `MCP2515_CS`  -> `PB6`
- `MCP2515_SO`  -> `PB4`
- `MCP2515_SI`  -> `PB5`
- `MCP2515_SCK` -> `PB3`
- `MCP2515_INT` -> `PB7`

### Используемый SPI
- `SPI3`
- `PB3 = SPI3_SCK`
- `PB4 = SPI3_MISO`
- `PB5 = SPI3_MOSI`
- `SPI3 prescaler = 8`
- рассчитанная скорость SPI3: **5.25 Мбит/с**

### DMA в проекте
В `.ioc` уже есть DMA для `SPI3`, но **новую базовую версию драйвера не усложнять DMA**.

DMA пригодится позже, но стартовая версия должна быть без DMA.

### EXTI
- `PB7` настроен как `GPIO_MODE_IT_FALLING`
- `PB7` должен быть с `GPIO_PULLUP`

### Таймер
В проекте есть `TIM2`, но в новой базовой версии heartbeat-драйвера таймер **не обязателен**.

## Модуль MCP2515/TJA1050

Подтверждённый состав модуля:
- `MCP2515`
- `TJA1050`

Критически важно:
- модуль должен быть запитан от **5 В**;
- без 5 В приём CAN не работал;
- SPI-регистры при этом могли читаться, что вводило в заблуждение.

## Шина CAN

### Текущая рабочая скорость
- `CAN bitrate = 250000`

### Текущий кварц MCP2515
- `MCP2515 oscillator = 8 МГц`

### Дополнительная информация
Ранее обсуждался переход на 16 МГц кварц, но **на текущем этапе исходить из реально установленного 8 МГц кварца**.

## ODrive

### Узлы
- `node_id_1 = 1`
- `node_id_2 = 2`

### Heartbeat
Сейчас ODrive настроены на heartbeat, и монитор CAN его видит.

Использовать:
- `cmd_id = 0x01`
- Heartbeat ID рассчитывается как `(node_id << 5) | 0x01`

Итого:
- `node 1` -> `0x21`
- `node 2` -> `0x41`

### Формат heartbeat
Нужно считать heartbeat как кадр с `DLC = 8`:
- bytes `0..3` -> `axis_error` (`uint32_t`, little-endian)
- byte `4` -> `axis_state`
- byte `5` -> `procedure_result`
- byte `6` -> `trajectory_done`
- byte `7` -> резерв/игнорировать

### Encoder Estimates
Это следующий этап, не первый.

Использовать позже:
- `cmd_id = 0x09`
- ID:
  - `node 1` -> `0x29`
  - `node 2` -> `0x41?` Нет, для encoder estimates node 2 -> `0x49`

Формат `Encoder Estimates`:
- `DLC = 8`
- bytes `0..3` -> `position_estimate` (`float32`, little-endian)
- bytes `4..7` -> `velocity_estimate` (`float32`, little-endian)

На текущем шаге **не реализовывать encoder estimates в первой базовой версии**, только оставить это в ТЗ как следующий этап.

## Что уже было подтверждено экспериментом

После подачи 5 В на модуль были получены такие признаки:
- `INT` начал работать;
- heartbeat начали приниматься;
- `last_id` принимал значение `0x41`;
- `last_dlc = 8`;
- счётчики heartbeat и прерываний росли.

Также было видно переполнение RX буфера при длинных `HAL_Delay(50)` в цикле. Значит:
- в основном цикле нельзя ставить длинные блокирующие задержки;
- приём MCP2515 нужно обслуживать быстро.

## Цель новой версии `can_mcp2515`

### Этап 1. Базовая версия
Сделать **простую стабильную версию** только для heartbeat.

Обязательно:
- init MCP2515;
- режим `NORMAL` по умолчанию для надёжной проверки на реальной шине;
- приём heartbeat двух ODrive;
- работа через polling;
- обработка EXTI от `INT`;
- хранение последних heartbeat-данных по двум узлам;
- минимальные диагностические поля.

### Этап 2. После подтверждения стабильности
Добавить:
- опциональный режим `LISTEN_ONLY`;
- приём `Encoder Estimates`;
- поля для `position` и `velocity`.

### Этап 3. Только потом
Обсуждать:
- активные запросы к ODrive;
- scheduler;
- DMA;
- расширенные timestamp и очереди.

## Принципиальные решения для новой версии

### 1. Не брать старую архитектуру sample FIFO как основу
Для heartbeat достаточно хранить:
- последнее состояние `hb_1`
- последнее состояние `hb_2`

Без FIFO на первом этапе.

### 2. Без DMA
Новая базовая версия должна работать через обычный blocking SPI:
- `HAL_SPI_Transmit`
- `HAL_SPI_TransmitReceive`

### 3. Без таймера и scheduler
На heartbeat таймер не нужен.

### 4. Без loopback на первом этапе
Нужен реальный приём heartbeat с шины.

### 5. Без float в базовой heartbeat-версии
Heartbeat-версия должна использовать только целочисленные поля.

## Последовательность инициализации

Новая версия должна делать такую последовательность:
1. Проверка параметров структуры.
2. Очистка runtime-полей.
3. `CS = 1`.
4. `RESET` MCP2515.
5. Переход в `CONFIG`.
6. Запись `CNF1/CNF2/CNF3`.
7. `RXB0CTRL = accept all`.
8. `RXB1CTRL = accept all`.
9. Очистка `CANINTF`.
10. Очистка `EFLG`.
11. Разрешение `RX0IE | RX1IE`.
12. Переход в выбранный режим, по умолчанию `NORMAL`.
13. Чтение назад `CANINTE` и `CANSTAT` в отладочные поля.
14. Установка `is_initialized = 1`.

## Поведение poll

`can_mcp2515_poll()` должен быть простым:
- если `int_pending == 0` и `INT` пин в high, сразу выход `OK`;
- иначе:
  - сбросить `int_pending`;
  - прочитать `CANSTAT`, `CANINTF`, `EFLG`;
  - если `RX0IF` установлен, прочитать RXB0;
  - если `RX1IF` установлен, прочитать RXB1;
  - если принят heartbeat `0x21`, обновить `hb_1`;
  - если принят heartbeat `0x41`, обновить `hb_2`;
  - очистить `RX0IF/RX1IF` через `BIT_MODIFY`.

## Важные замечания по циклу main

В `while(1)` нельзя делать длинные блокировки после факта приёма heartbeat.

Запрещено на первом этапе:
- `HAL_Delay(50)` внутри реакции на heartbeat;
- любые долгие процедуры после каждого кадра.

Разрешено:
- короткий `HAL_Delay(1)` в конце цикла или вовсе без него.

## Требования к примеру использования в `.h`

В заголовочном файле сразу после include должен быть короткий пример:
- заполнение структуры;
- `hspi = &hspi3`;
- `cs_port = GPIOB`, `cs_pin = GPIO_PIN_6`;
- `int_port = GPIOB`, `int_pin = GPIO_PIN_7`;
- `mcp2515_osc_hz = 8000000U`;
- `can_bitrate = 250000U`;
- `node_id_1 = 1U`;
- `node_id_2 = 2U`;
- `mode = CAN_MCP2515_MODE_NORMAL`;
- вызов `can_mcp2515_init()`;
- вызов `can_mcp2515_poll()` в `while(1)`.

## Что НЕ делать в новой версии

На старте не делать:
- DMA;
- FIFO encoder samples;
- scheduler запросов;
- loopback test;
- активную передачу RTR;
- float parsing для encoder;
- работу через таймер;
- сложный state machine.

## Критерий готовности новой версии

Новая базовая версия считается успешной, если:
1. собирается в текущем проекте `ICMDriver`;
2. инициализирует MCP2515 без ошибки;
3. на реальной шине с двумя ODrive при `250000` принимает heartbeat;
4. `int_count` растёт;
5. `hb_count` растёт;
6. `hb_1.valid` и `hb_2.valid` становятся ненулевыми;
7. `hb_1.axis_state` и `hb_2.axis_state` обновляются;
8. нет постоянного overflow, вызванного самим кодом main loop.

## Краткая стратегия разработки в новом чате

В новом чате просить модель сделать **новую базовую версию `can_mcp2515`**, не опираясь напрямую на старый большой драйвер, а используя только подтверждённые рабочие решения:
- blocking SPI;
- EXTI по `INT`;
- `NORMAL` mode;
- heartbeat only;
- accept all RX buffers;
- минимум логики;
- UTF-8 комментарии;
- версия в формате `xx.yy` и дата/время в формате `dd.mm.yy hh:mm:ss` в заголовке `.h`.
